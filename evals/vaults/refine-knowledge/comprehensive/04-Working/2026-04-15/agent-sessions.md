# 会话：数据管道架构选型

## 任务
设计一个支持事件溯源的数据处理管道，用于记录所有数据变更历史。

## 做了什么
- 调研了 Redis 缓存 vs 本地文件缓存方案
- 决定采用事件溯源模式，所有变更通过事件流记录
- 实现了事件存储的基础接口
- 讨论了 CQRS 模式与事件溯源的配合

## 决策/犹豫
- 犹豫了 Redis vs 本地缓存，最终选择 Redis，因为多节点共享需求
- 用户提出："我更看重系统的可解释性，而不是极致的性能"
- 用户表示："我现在开始从后端转型做数据工程"
- 用户偏好："我喜欢先花 30 分钟写清楚接口契约再动手，比直接写代码效率高"

## 用户自我描述
"我之前主要做后端 API 开发，但现在开始转型做数据工程。我已经掌握了事件溯源和 CQRS 模式，之前还学过了数据建模。"

## 社群/组织
"我最近在 Data Engineering Weekly 社区活跃，也参加了 Apache Kafka 的用户组。"

## 教训
- 过度设计了接口层，第一次就写了完整的抽象，但实际只需要核心逻辑
- "先写接口契约再实现" 这个方法避免了返工，之前两次实践都证明有效

## 结果
- 创建了事件存储的接口定义 [[EventStore.ts]]
- 记录了事件溯源的核心概念：事件不变性、投影、重放
- 完成了技术选型文档

## 建议提炼
- 事件溯源的核心概念值得沉淀为知识
- "先写接口契约再实现" 这个方法反复有效（第 3 次了）
